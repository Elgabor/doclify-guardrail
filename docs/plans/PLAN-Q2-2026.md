## Contesto

Doclify v1.7.2. Core CLI completo: 35 regole, 14 auto-fix, health score, trending locale, GitHub Action.
Il CLI è gratuito e lo resta. L'obiettivo è costruire le feature che giustificano un tier paid a $15-30/seat/mese.

## Insight dalla ricerca competitiva

**Swimm** ha pivotato a pricing per righe di codice (non più per seat). Si è spostato verso "AI coding assistant" che genera documentazione, allontanandosi dal quality enforcement. La sua debolezza resta: nessun linting, nessun quality score, nessun link checking.

**SonarQube** ha consolidato il modello quality gate + dashboard + trending + CI blocking + notifiche. Il loro "Sonar way" è un gate di default read-only con condizioni sul nuovo codice. Questo approccio "Clean as You Code" è direttamente trasferibile ai docs: "Clean as You Doc" — mantieni la qualità sul nuovo contenuto senza dover sistemare tutto il pregresso.

**Il mercato doc tools** cresce verso $12.5B entro 2033. I tool AI-driven dovrebbero catturare oltre il 25% del mercato entro fine 2026. Il gap resta: nessun tool combina quality gate + scoring + enforcement in CI per documentazione.

**SonarQube MCP Server** — SonarQube ha già un MCP server che permette ad agenti AI (Claude Code, Cursor, etc.) di leggere issue e quality gate status. Questo crea un precedente: il modello "quality gate accessibile da agenti AI" è la direzione del mercato.

## Feature prioritizzate per rapporto impatto/effort

### P0 — Sblocca il tier paid (Q2)

#### 1. Dashboard Cloud + Score API
**Effort: L | Impatto: CRITICO | Sblocca pricing: SÌ**

Il trending locale (ASCII) non basta per i team. La feature che i team pagano è la **visibilità condivisa**.

Cosa serve:
- API REST per push dei risultati di analisi (POST /v1/scores)
- Dashboard web con storico per progetto, per file, per regola
- Trend line nel tempo con alert su regressioni
- Webhook per notifiche (Slack, email, custom)
- Badge SVG dinamico servito dalla dashboard (non generato localmente)

Modello di riferimento: SonarQube dashboard con quality gate status, trend, e PR decoration.

Decisione aperta: self-hosted (open source dashboard) o SaaS puro?
Raccomandazione: SaaS puro per v1. Riduce effort e abilita pricing ricorrente.

#### 2. Quality Gate "Clean as You Doc"
**Effort: M | Impatto: ALTO | Sblocca pricing: SÌ (in combinazione con Dashboard)**

Adotta il principio SonarQube "Clean as You Code" per i docs: misura la qualità solo sui file nuovi o modificati, non su tutto il pregresso. Questo riduce la barriera di adozione.

Cosa serve:
- `doclify --gate` che distingue "new docs" da "overall"
- Condizioni configurabili: min-score per nuovi file, max-issues per nuovi file
- Gate status nel commento PR: PASS/FAIL con dettaglio
- Notifica su quality gate change (webhook → Slack)
- Gate di default "Doclify way" (read-only, come "Sonar way")

Il CLI ha già `--min-score` e `--diff`. Questa feature li unifica in un concetto coerente di "gate" con condizioni multiple e configurabili.

### P1 — Differenziante AI (Q2-Q3)

#### 3. AI Drift Guard v2 — Code-Aware
**Effort: M | Impatto: ALTO | Sblocca pricing: SÌ (feature AI del tier paid)**

Il drift guard v1 è offline e generico. V2 deve sapere quale codice è cambiato e quale doc dovrebbe aggiornarsi.

Cosa serve:
- Mapping bidirezionale code↔doc (per path + per frontmatter)
- Integrazione git diff: se src/score.js cambia, flag docs/scoring.md
- Alert mirato nel commento PR: "questi docs potrebbero essere disallineati"
- Nessuna dipendenza esterna (usa child_process per git, embedding locale opzionale)

Differenziatore vs Swimm: Swimm fa auto-sync (riscrive i docs). Doclify fa quality enforcement (ti dice che i docs sono disallineati e ti blocca se non li aggiorni). Approcci complementari, non sostitutivi.

#### 4. AI Auto-Review su PR
**Effort: M | Impatto: MEDIO-ALTO | Sblocca pricing: SÌ**

Quando una PR modifica file .md, Doclify analizza il contenuto e lascia commenti inline:
- "Questo paragrafo ha un placeholder non compilato"
- "Il link a /api/v2 non esiste nel repo"
- "Questo doc non è aggiornato da 8 mesi e stai modificando il codice correlato"

Cosa serve:
- GitHub App (non solo Action) per permessi di commento inline
- Logica di review che combina lint + link check + freshness + drift
- Commento strutturato: summary con score + commenti inline sui problemi

### P2 — Estensibilità (Q3)

#### 5. Custom Rules Builder
**Effort: M | Impatto: MEDIO | Sblocca pricing: NO (ma aumenta retention)**

Permetti ai team di definire regole custom in YAML/JS senza forkare il progetto.

Cosa serve:
- File `.doclify/rules/` con regole custom per progetto
- Formato: YAML per regole semplici (pattern match), JS per regole complesse
- Le regole custom partecipano al calcolo dello score
- Documentazione ed esempi per i casi più comuni

Modello di riferimento: Vale con i suoi "styles" custom. Ma più semplice.

#### 6. Multi-Repo Dashboard
**Effort: S-M | Impatto: MEDIO | Sblocca pricing: SÌ (feature enterprise)**

Un'unica dashboard per tutti i repo dell'organizzazione. Score aggregato, trend comparativi, regole condivise.

Prerequisito: Dashboard Cloud (feature 1) già implementata.

| Plugin IDE | Effort alto, valore marginale. Il valore è in CI, non nell'editor |
| Hosting docs | ReadMe/Mintlify lo fanno. Noi siamo il quality gate, non la piattaforma |
